1. Schema vs Model – What's the Difference?
Schema defines the STRUCTURE, Model provides the INTERFACE
Schema → Blueprint of document (fields, types, validation)
Model → Constructor that creates documents and queries database
// Schema: Defines the shape of the data const userSchema = new mongoose.Schema({ name: String, email: String, age: Number }); // Model: Creates the collection and provides methods const User = mongoose.model('User', userSchema); // Now you can do: // User.find(), User.create(), User.findById(), etc.

5. Virtuals, Instance Methods & Static Methods
const userSchema = new mongoose.Schema({ firstName: { type: String, required: true }, lastName: { type: String, required: true }, birthDate: Date, email: String, password: String, posts: [{ type: mongoose.Schema.Types.ObjectId, ref: 'Post' }] }); // ========== VIRTUAL FIELDS (not stored in DB) ========== // Virtual getter userSchema.virtual('fullName').get(function() { return `${this.firstName} ${this.lastName}`; }); // Virtual setter userSchema.virtual('name').set(function(name) { const parts = name.split(' '); this.firstName = parts[0]; this.lastName = parts[1]; }); userSchema.virtual('age').get(function() { if (!this.birthDate) return null; const today = new Date(); const birth = new Date(this.birthDate); let age = today.getFullYear() - birth.getFullYear(); const m = today.getMonth() - birth.getMonth(); if (m < 0 || (m === 0 && today.getDate() < birth.getDate())) { age--; } return age; }); // Virtual for populated posts userSchema.virtual('postCount').get(function() { return this.posts?.length || 0; }); // ========== INSTANCE METHODS (methods on document) ========== userSchema.methods.getProfile = function() { return { name: this.fullName, email: this.email, age: this.age }; }; userSchema.methods.comparePassword = async function(candidatePassword) { return await bcrypt.compare(candidatePassword, this.password); }; // ========== STATIC METHODS (methods on model) ========== userSchema.statics.findByEmail = function(email) { return this.findOne({ email }); }; userSchema.statics.getActiveUsers = function() { return this.find({ isActive: true }); }; // ========== QUERY HELPERS ========== userSchema.query.byName = function(name) { return this.where({ name: new RegExp(name, 'i') }); }; userSchema.query.active = function() { return this.where({ isActive: true }); }; // Usage: // const user = await User.findById(id); // user.fullName → virtual // user.getProfile() → instance method // await User.findByEmail('test@example.com') → static method // await User.find().byName('Ramesh').active() → query helper
6. Indexing for Performance
const productSchema = new mongoose.Schema({ name: { type: String, required: true }, price: { type: Number, required: true }, category: String, tags: [String], createdAt: Date, rating: Number }); // Single field index productSchema.index({ name: 1 }); // 1 = ascending, -1 = descending // Compound index (multiple fields) productSchema.index({ category: 1, price: -1 }); // Text index for search productSchema.index({ name: 'text', description: 'text' }); // Unique index productSchema.index({ email: 1 }, { unique: true }); // Sparse index (only for documents that have the field) productSchema.index({ sku: 1 }, { sparse: true }); // Expiring documents (TTL index – auto-delete after time) productSchema.index({ createdAt: 1 }, { expireAfterSeconds: 3600 }); // 1 hour // Geospatial index (for location queries) productSchema.index({ location: '2dsphere' }); // Partial index (only index certain documents) productSchema.index( { price: 1 }, { partialFilterExpression: { price: { $gt: 100 } } } ); // Background indexing (doesn't block other operations) productSchema.index({ createdAt: -1 }, { background: true }); // Check existing indexes in MongoDB Shell: // db.products.getIndexes() // db.products.dropIndex('index_name')
7. Subdocuments vs Referenced Documents (Relationships)
📦 Subdocuments (Embedded)
// Embed related data directly in the parent document const orderSchema = new mongoose.Schema({ customer: { name: String, email: String, address: { street: String, city: String } }, items: [{ productName: String, quantity: Number, price: Number }], total: Number }); // Subdocument methods orderSchema.methods.calculateTotal = function() { this.total = this.items.reduce((sum, item) => sum + (item.price * item.quantity), 0); return this.total; }; // When to embed: // ✅ Data is always accessed together // ✅ Data has one-to-one or one-to-few relationship // ✅ Data doesn't change frequently
🔗 Referenced Documents (using ObjectId)
// models/User.js const userSchema = new mongoose.Schema({ name: String, email: String, posts: [{ type: mongoose.Schema.Types.ObjectId, ref: 'Post' }] }); // models/Post.js const postSchema = new mongoose.Schema({ title: String, content: String, author: { type: mongoose.Schema.Types.ObjectId, ref: 'User' }, comments: [{ type: mongoose.Schema.Types.ObjectId, ref: 'Comment' }] }); // models/Comment.js const commentSchema = new mongoose.Schema({ text: String, author: { type: mongoose.Schema.Types.ObjectId, ref: 'User' }, post: { type: mongoose.Schema.Types.ObjectId, ref: 'Post' } }); // Populating references (JOIN-like operation) const posts = await Post.find() .populate('author', 'name email') // Populate author with specific fields .populate({ path: 'comments', populate: { path: 'author', select: 'name' } }); // When to reference: // ✅ Data grows independently (unlimited comments) // ✅ Data needs to be updated frequently // ✅ Many-to-many relationships

9. Schema Design Best Practices
✅ DO's

• Always add timestamps
• Use appropriate data types
• Add indexes for frequently queried fields
• Use meaningful field names
• Add validation rules
• Use virtuals for computed fields
• Add unique indexes for email/username

❌ DON'Ts

• Don't embed unlimited arrays (max 16MB document)
• Don't store passwords as plain text
• Don't create too many indexes
• Don't use Mixed type unless necessary
• Don't skip validation
• Don't store large files in MongoDB (use GridFS or S3)

Schema Design Patterns:
• **One-to-One** → Embedded (preferred) or Reference
• **One-to-Many (few)** → Embedded array
• **One-to-Many (many)** → Reference with parent ID
• **Many-to-Many** → Reference array or junction collection
