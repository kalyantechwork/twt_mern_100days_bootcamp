Relationships – Population & Referencing

Master MongoDB relationships! Learn how to connect collections using references, populate data like SQL JOINs, and design efficient data models for your MERN applications.

Telugu WebTech · Master MongoDB Relationships
1. Types of Relationships in MongoDB
1. One-to-One (1:1)

A user has one profile. A product has one SKU.

// Embedded (preferred) { user: { name: "Ramesh", profile: { bio: "...", avatar: "..." } } }
2. One-to-Many (1:N)

A user has many posts. A category has many products.

// Reference (preferred for many) { posts: [{ type: ObjectId, ref: 'Post' }] }
3. Many-to-Many (N:N)

Users belong to many groups. Products have many tags.

// Two-way references { users: [{ type: ObjectId, ref: 'User' }], groups: [{ type: ObjectId, ref: 'Group' }] }
4. One-to-Few (1:N with small N)

A blog post has few comments (under 100).

// Embedded array (preferred) { comments: [{ text: "...", author: "..." }] }
Design Rule of Thumb:
• Embed when data is always accessed together and has few items
• Reference when data grows independently or is accessed separately
2. Setting Up Reference Schemas (ObjectId)
// models/User.js const mongoose = require('mongoose'); const userSchema = new mongoose.Schema({ name: { type: String, required: true }, email: { type: String, required: true, unique: true }, bio: String, avatar: String, createdAt: { type: Date, default: Date.now } }, { timestamps: true }); // models/Post.js const postSchema = new mongoose.Schema({ title: { type: String, required: true }, content: { type: String, required: true }, // Reference to User model author: { type: mongoose.Schema.Types.ObjectId, ref: 'User', required: true }, likes: [{ type: mongoose.Schema.Types.ObjectId, ref: 'User' }], createdAt: { type: Date, default: Date.now } }, { timestamps: true }); // models/Comment.js const commentSchema = new mongoose.Schema({ text: { type: String, required: true }, // Reference to User author: { type: mongoose.Schema.Types.ObjectId, ref: 'User', required: true }, // Reference to Post post: { type: mongoose.Schema.Types.ObjectId, ref: 'Post', required: true }, // Self-reference for nested comments parentComment: { type: mongoose.Schema.Types.ObjectId, ref: 'Comment' }, likes: [{ type: mongoose.Schema.Types.ObjectId, ref: 'User' }] }, { timestamps: true }); const User = mongoose.model('User', userSchema); const Post = mongoose.model('Post', postSchema); const Comment = mongoose.model('Comment', commentSchema); module.exports = { User, Post, Comment };

4. Basic Population (Replacing IDs with Documents)
// Without population – you only get IDs const postWithoutPopulate = await Post.findById(postId); console.log(postWithoutPopulate.author); // ObjectId("507f1f77bcf86cd799439011") // WITH population – replaces ID with actual document const postWithPopulate = await Post.findById(postId).populate('author'); console.log(postWithPopulate.author); /* Output: { _id: ObjectId("..."), name: "Ramesh Kumar", email: "ramesh@example.com", bio: "Full Stack Developer", avatar: "https://example.com/avatar.jpg" } */ // Populate multiple fields const postWithMultiple = await Post.findById(postId) .populate('author') .populate('likes'); // Populate with specific fields (only include needed data) const postWithSelect = await Post.findById(postId) .populate('author', 'name email avatar') // Only these fields .populate('likes', 'name'); // Only name field // Populate nested references (comments then comment authors) const postWithComments = await Post.findById(postId) .populate('author') .populate({ path: 'comments', populate: { path: 'author', select: 'name avatar' } });

7. Embedded vs Referenced – When to Use What?
📦 EMBEDDED (Store inside parent)

Use when:
• Data is always accessed together
• One-to-one or one-to-few relationships
• Data doesn't change frequently
• Need atomic updates
• Data size is small (under 16MB)

{ user: { name: "Ramesh", address: { city: "HYD", zip: "500001" } } }
🔗 REFERENCED (Store ObjectId)

Use when:
• Data grows independently
• One-to-many or many-to-many relationships
• Need to query related data separately
• Data is updated frequently
• Documents are large or unlimited

{ post: { title: "My Post", authorId: ObjectId("...") } }
🔄 Hybrid Approach (Best of Both Worlds)
// Store recent comments embedded, older comments referenced const postSchema = new mongoose.Schema({ title: String, content: String, recentComments: [{ text: String, author: String, createdAt: Date }], // Embedded (for quick access) commentCount: Number, // Denormalized count commentsRef: [{ type: ObjectId, ref: 'Comment' }] // Referenced (for all comments) }); // Update commentCount automatically postSchema.pre('save', function(next) { this.commentCount = this.recentComments.length; next(); });
8. Performance Tips for Population
✅ DO's

• Add indexes on foreign keys (author, post references)
• Use select to only populate needed fields
• Use lean() when you don't need Mongoose documents
• Limit population depth (avoid circular populations)
• Use pagination with populate (limit + skip)

❌ DON'Ts

• Don't populate everything in a list view
• Don't create deep nested populations (> 3 levels)
• Don't populate in loops (use Model.populate() instead)
• Don't store large arrays of references (use pagination)

// BAD: Populating in loop (N+1 query problem) const users = await User.find(); for (const user of users) { await user.populate('posts'); // One query per user! } // GOOD: Populate all at once const users = await User.find().populate('posts'); // BAD: Populating everything const posts = await Post.find().populate('author').populate('comments'); // GOOD: Selective population for list view const posts = await Post.find() .populate('author', 'name') // Only name needed .select('title createdAt'); // Only post title and date // Using lean() for read-only operations (2-3x faster) const posts = await Post.find().populate('author').lean();
