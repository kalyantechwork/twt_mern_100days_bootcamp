Welcome to MongoDB! The most popular NoSQL database for modern web applications. Learn why MongoDB is perfect for MERN stack development.

1. What is MongoDB?

MongoDB is a NoSQL, document-oriented database that stores data in flexible, JSON-like documents (BSON). Unlike traditional SQL databases, MongoDB doesn't use tables and rows – it uses collections and documents.

MongoDB vs SQL Terminology:
SQL: Database → Table → Row → Column
MongoDB: Database → Collection → Document → Field

SQL Database

• Fixed schema (structure required)
• Tables with rows and columns
• Uses JOINs for related data
• Scales vertically
• Examples: MySQL, PostgreSQL

MongoDB (NoSQL)

• Flexible schema (dynamic)
• Collections with documents
• Embedded documents for relationships
• Scales horizontally
• Example: MongoDB

2. Why MongoDB for MERN Stack?

👥 Developer Friendly

Easy to learn, great documentation, huge community support.

MERN = MongoDB + Express + React + Node.js – They work together seamlessly because all use JavaScript/JSON!
3. Understanding MongoDB Data Model
📄 Document (Like a JSON Object)
{ "_id": ObjectId("507f1f77bcf86cd799439011"), "name": "Ramesh", "email": "ramesh@example.com", "age": 25, "hobbies": ["coding", "reading", "traveling"], "address": { "street": "MG Road", "city": "Hyderabad", "pincode": 500001 }, "createdAt": ISODate("2024-01-15T10:30:00Z"), "isActive": true }
📁 Collection (Like a Table)
// Users Collection - stores multiple user documents users = [ { _id: 1, name: "Ramesh", email: "ramesh@example.com" }, { _id: 2, name: "Priya", email: "priya@example.com" }, { _id: 3, name: "Suresh", email: "suresh@example.com" } ] // Different documents can have different fields (flexible schema!) users = [ { _id: 1, name: "Ramesh", email: "ramesh@example.com", age: 25 }, { _id: 2, name: "Priya", email: "priya@example.com", phone: "9876543210" }, { _id: 3, name: "Suresh", email: "suresh@example.com", address: "Delhi" } ]
Important: Each document has a unique _id field (ObjectId) automatically created by MongoDB.

5. Handling Relationships in MongoDB

Unlike SQL, MongoDB has two ways to handle relationships between data:

📦 1. Embedded Documents (Denormalization)
// Store related data inside the same document { "_id": 1, "title": "My Blog Post", "content": "Post content here...", "author": { "name": "Ramesh", "email": "ramesh@example.com" }, "comments": [ { "user": "Priya", "text": "Great post!", "date": "2024-01-15" }, { "user": "Suresh", "text": "Very helpful", "date": "2024-01-16" } ] } // ✅ Good for: One-to-One, One-to-Few relationships // ✅ Benefit: Single query gets all data
🔗 2. Referenced Documents (Normalization)
// Store references (IDs) to related documents // users collection { "_id": 1, "name": "Ramesh", "email": "ramesh@example.com" } // posts collection { "_id": 100, "title": "My Post", "authorId": 1, "content": "..." } // comments collection { "_id": 200, "postId": 100, "userId": 2, "text": "Nice post!" } // ✅ Good for: One-to-Many, Many-to-Many relationships // ✅ Benefit: No data duplication, easier updates
Best Practice: Embed when data is always accessed together. Reference when data grows independently or needs frequent updates.
6. Basic MongoDB Operations (CRUD)
📝 Create (Insert)
// Insert one document db.users.insertOne({ name: "Ramesh", email: "ramesh@example.com", age: 25 }) // Insert multiple documents db.users.insertMany([ { name: "Priya", email: "priya@example.com" }, { name: "Suresh", email: "suresh@example.com" } ])
📖 Read (Query)
// Find all users db.users.find() // Find with filter db.users.find({ age: { $gt: 18 } }) // age > 18 // Find one document db.users.findOne({ email: "ramesh@example.com" }) // Find with projection (select fields) db.users.find({}, { name: 1, email: 1, _id: 0 })
✏️ Update
// Update one document db.users.updateOne( { name: "Ramesh" }, { $set: { age: 26 } } ) // Update multiple documents db.users.updateMany( { age: { $exists: false } }, { $set: { age: 18 } } ) // Replace entire document db.users.replaceOne( { name: "Ramesh" }, { name: "Ramesh Kumar", email: "ramesh.k@example.com", age: 27 } )
🗑️ Delete
// Delete one document db.users.deleteOne({ name: "Suresh" }) // Delete multiple documents db.users.deleteMany({ age: { $lt: 18 } }) // Delete all documents db.users.deleteMany({})
7. MongoDB Query Operators
Comparison Operators

$eq = equal to
$gt > greater than
$gte >= greater than or equal
$lt < less than
$lte <= less than or equal
$ne != not equal

Logical Operators

$and – all conditions true
$or – at least one true
$nor – none true
$not – negates condition

Array Operators

$in – value in array
$nin – value not in array
$all – matches all values
$size – array length

Element Operators

$exists – field exists
$type – field type

// Query examples db.users.find({ age: { $gte: 18, $lte: 60 } }) // Age between 18-60 db.users.find({ $or: [{ age: 25 }, { name: "Ramesh" }] }) db.users.find({ hobbies: { $in: ["coding", "reading"] } }) db.users.find({ address: { $exists: true } })
8. MongoDB vs SQL – When to Use Which?
✅ Choose MongoDB when:

• Schema is flexible or changes frequently
• Data is JSON-like (documents)
• Need horizontal scaling
• Fast development and iteration
• Building modern web/mobile apps
• Working with MERN stack

✅ Choose SQL when:

• Data has strict, fixed schema
• Need complex JOIN operations
• ACID transactions are critical
• Reporting and analytics are primary
• Working with existing SQL infrastructure
