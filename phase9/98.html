Continuous Integration · Continuous Deployment · GitHub Actions · Auto-Deploy

1️⃣ What is CI/CD? (Devops Essential)

CI/CD stands for Continuous Integration and Continuous Deployment — automating the process from code commit to production deployment.

Code Commit → CI (Test & Build) → CD (Deploy) → Production
💡 Analogy: CI/CD is like an automated assembly line. Write code → push to GitHub → automatically tested → automatically deployed → users see updates instantly!

🔄 Continuous Integration (CI)

• Automatically test code on every push
• Run linters, unit tests, build checks
• Catch bugs early before merge
• Every developer pushes daily

🚀 Continuous Deployment (CD)

• Automatically deploy after successful tests
• Push to staging or production
• Eliminate manual deployment steps
• Rollback if issues detected

3️⃣ Why CI/CD? (Benefits for Your MERN App)
  • ✅ No more manual deployments — git push = auto-deploy
  • ✅ Catch bugs early — automated tests run on every commit
  • ✅ Team collaboration — all code is tested before merging
  • ✅ Consistent process — same steps every time
  • ✅ Quick rollback — revert to previous version instantly
4️⃣ GitHub Actions — Free CI/CD for GitHub Repos
# GitHub Actions workflow file location:
.github/workflows/main.yml

# Structure:
name: CI/CD Pipeline
on: [push, pull_request]
jobs:
build:
runs-on: ubuntu-latest
steps:
- name: Checkout code
uses: actions/checkout@v3
- name: Install dependencies
run: npm ci
- name: Run tests
run: npm test
- name: Build
run: npm run build
🤖 Free for public repos! GitHub Actions gives you 2000 free minutes/month for private repos, unlimited for public.
5️⃣ Backend CI/CD — Test & Deploy to Render
# .github/workflows/backend.yml
name: Backend CI/CD

on:
push:
branches: [main]
paths: ['backend/**']
pull_request:
branches: [main]

jobs:
test-and-deploy:
runs-on: ubuntu-latest
steps:
- uses: actions/checkout@v3
- name: Setup Node.js
uses: actions/setup-node@v3
with:
node-version: '20'
- name: Install dependencies
run: cd backend && npm ci
- name: Run tests
run: cd backend && npm test
- name: Deploy to Render
if: github.ref == 'refs/heads/main'
run: curl -X POST ${{ secrets.RENDER_DEPLOY_HOOK_URL }}
6️⃣ Frontend CI/CD — Deploy to Vercel
# .github/workflows/frontend.yml
name: Frontend CI/CD

on:
push:
branches: [main]
paths: ['frontend/**']
pull_request:
branches: [main]

jobs:
build-and-deploy:
runs-on: ubuntu-latest
steps:
- uses: actions/checkout@v3
- name: Setup Node.js
uses: actions/setup-node@v3
with:
node-version: '20'
- name: Install dependencies
run: cd frontend && npm ci
- name: Lint
run: cd frontend && npm run lint
- name: Build
run: cd frontend && npm run build
- name: Deploy to Vercel
uses: amondnet/vercel-action@v20
with:
vercel-token: ${{ secrets.VERCEL_TOKEN }}
vercel-org-id: ${{ secrets.VERCEL_ORG_ID }}
vercel-project-id: ${{ secrets.VERCEL_PROJECT_ID }}
vercel-args: '--prod'
7️⃣ GitHub Secrets — Store Sensitive Data
# Go to GitHub Repository → Settings → Secrets and variables → Actions
# Add these secrets:

RENDER_DEPLOY_HOOK_URL = https://api.render.com/deploy/...
VERCEL_TOKEN = your_vercel_api_token
VERCEL_ORG_ID = team_xxxxxxxxxx
VERCEL_PROJECT_ID = prj_xxxxxxxxxx
MONGODB_URI = mongodb+srv://...
JWT_SECRET = your_secret_key

9️⃣ Complete MERN CI/CD Pipeline
🔄 CI/CD Pipeline Simulation:
📝 Developer pushes code to GitHub (main branch)
⚡ GitHub Actions triggers workflow
📦 Checkout code + Install dependencies (20s)
🧪 Run tests (5s) → ✅ All tests passed
🔨 Build frontend (15s) → ✅ Build successful
🚀 Deploy backend to Render (10s) → ✅ Live
🌐 Deploy frontend to Vercel (15s) → ✅ Live
✅ Pipeline complete! App updated in ~60 seconds
🔟 CI/CD Best Practices

🔀 Small, Frequent Commits

Push daily, not weekly — catch issues early

✅ Always Pass Tests

Never merge failing tests to main

🔒 Secrets Management

Never hardcode secrets — use GitHub Secrets

🚦 Branch Protection

Require CI to pass before merging PRs

1️⃣1️⃣ Branch Protection — Enforce Quality
# GitHub Repository → Settings → Branches → Add rule
# Branch name pattern: main
# Require status checks to pass before merging:
# • CI / build (GitHub Actions)
# • Vercel / Netlify preview deployment
# Require pull request reviews (at least 1)
# Dismiss stale reviews when new commits pushed
1️⃣2️⃣ Rollback — Revert to Previous Version
# Option 1: Revert the commit
git revert COMMIT_HASH
git push origin main

# CI/CD will auto-deploy the reverted version

# Option 2: Redeploy previous build (Vercel)
# Dashboard → Deployments → Find previous version → "Promote to Production"

⚠️ Common CI/CD Mistakes
  • ❌ Not using environment secrets → exposing API keys in code
  • ❌ Running expensive tests on every push → use path filters
  • ❌ No branch protection → broken code can be merged
  • ❌ No rollback plan → can't fix failed deployments quickly
💼 Common Interview Questions

❓ Why branch protection?

Prevents merging code that breaks tests or hasn't been reviewed.
