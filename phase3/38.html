1️⃣ What is Context API? (Global State Solution)

📌 Purpose & Usage

Context API is a React feature that allows you to share data across multiple components without passing props manually at every level (solving the "prop drilling" problem). It provides a way to pass data through the component tree without having to pass props down manually at every level.

Where it's used: Theme (dark/light mode), User authentication status, Language preferences, Shopping cart state, User settings, Notifications — any data that needs to be accessed by many components at different levels.

Without Context: App → Parent → Child → GrandChild → Data (prop drilling nightmare)
With Context: App (Provider) → Any Component (Consumer) → Direct Access!
2️⃣ The Problem: Prop Drilling
// Without Context — props must pass through every level
function App() { const [user, setUser] = useState("Kalyan"); return <Header user={user} />; }
function Header({ user }) { return <Navbar user={user} />; }
function Navbar({ user }) { return <div>Welcome, {user}</div>; }
// Header component didn't need 'user', but had to pass it down!
🧠 Prop Drilling Problem:
• Data passes through components that don't need it
• Makes code harder to maintain and refactor
• Components become tightly coupled
• Context API solves this by making data globally accessible!
3️⃣ The Solution: Context API in 3 Steps
// Step 1: Create Context
import { createContext, useState } from 'react';
export const UserContext = createContext();

// Step 2: Provide value (wrap components that need access)
function App() {
const [user, setUser] = useState("Kalyan");
return (<UserContext.Provider value={{ user, setUser }}><Header /></UserContext.Provider>);
}

// Step 3: Consume anywhere!
function Navbar() {
const { user } = useContext(UserContext);
return <div>Welcome, {user}</div>;
}
✅ Context API in 3 steps:
1. createContext() — creates a context object
2. Provider — wraps components and provides value
3. useContext() — consumes the context value
💡 Any component can access this data directly — no prop drilling!
4️⃣ Live Demo: Theme Context (Light/Dark Mode)
// ThemeContext.jsx
const ThemeContext = createContext();
export function useTheme() { return useContext(ThemeContext); }

export function ThemeProvider({ children }) {
const [theme, setTheme] = useState('light');
const toggleTheme = () => setTheme(prev => prev === 'light' ? 'dark' : 'light');
return (<ThemeContext.Provider value={{ theme, toggleTheme }}>{children}</ThemeContext.Provider>);
}

5️⃣ Real Example: User Authentication Context
const AuthContext = createContext();

function AuthProvider({ children }) {
const [user, setUser] = useState(null);
const login = (userData) => setUser(userData);
const logout = () => setUser(null);
return (<AuthContext.Provider value={{ user, login, logout }}>{children}</AuthContext.Provider>);
}

// Any component can now access user, login, logout
function Profile() { const { user, logout } = useContext(AuthContext); return ...; }

✅ Auth state (user login status) is global — any component can check if user is logged in
✅ Navbar, Profile, Dashboard, Checkout can all access this data directly
💡 This is exactly how real authentication works in production apps!
6️⃣ Shopping Cart Context Example
const CartContext = createContext();

function CartProvider({ children }) {
const [cart, setCart] = useState([]);
const addToCart = (product) => setCart(prev => [...prev, product]);
const total = cart.reduce((sum, item) => sum + item.price, 0);
return (<CartContext.Provider value={{ cart, addToCart, total }}>{children}</CartContext.Provider>);
}

✅ Product list, Cart icon, Checkout page can all access the same cart data
💡 This is how real e-commerce sites manage shopping carts!
7️⃣ Best Practice: Custom Hook for Context
// Create custom hook for cleaner consumption
export function useAuth() {
const context = useContext(AuthContext);
if (!context) throw new Error('useAuth must be used within AuthProvider');
return context;
}

// Usage in components — clean and simple!
function Profile() { const { user, logout } = useAuth(); return ...; }
✅ Custom hooks provide better error messages (fail early if used outside provider)
✅ Cleaner component code — useAuth() vs useContext(AuthContext)
✅ Easier to refactor — change context implementation without changing components
💡 This is the recommended pattern in professional React codebases!

⚠️ Common Context API Mistakes
  • ❌ Forgetting to wrap components with <Provider> → context undefined, components can't access data
  • ❌ Using context for every piece of state (only for truly global state that many components need)
  • ❌ Not separating context into multiple files → messy, hard to maintain code
  • ❌ Putting expensive computations in context value (causes unnecessary re-renders)
  • ❌ Not creating custom hooks → no error messages when used outside provider
💼 Common Interview Questions

❓ What is Context API?

React feature for global state management without prop drilling.

❓ When to use Context?

Theme, authentication, language, user preferences — data needed by many components.

❓ Context vs Redux?

Context is built-in, simpler. Redux has devtools, better for complex state and frequent updates.

❓ What is prop drilling?

Passing props through multiple components that don't need them — makes code messy.
