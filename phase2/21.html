1️⃣ What is Git? (Version Control System)

📌 Purpose & Usage

Git is a Version Control System (VCS) that tracks changes in your code — like a time machine for your project. Every tech company (Google, Microsoft, Amazon, Netflix, Meta) uses Git daily. It allows you to save snapshots of your code, revert to previous versions, and collaborate with teams without overwriting each other's work.

Where it's used: Every professional software project — from small personal projects to massive enterprise applications with thousands of developers.

Code Changes → Git tracks snapshots (commits) → View history → Restore any version
2️⃣ Install Git & Verify

📌 Installation Steps

Download Git from git-scm.com and install with default settings. After installation, verify by opening terminal/command prompt and running git --version.

# Download from https://git-scm.com/
# Install with default settings

# Verify installation (open terminal/command prompt)
git --version
$ git --version
git version 2.43.0

✅ If you see a version number, Git is installed successfully!
⚠️ If not, download from git-scm.com and reinstall.
3️⃣ Configure Git (Your Identity)

📌 Why Configure?

Git uses your name and email to track who made each commit. This information becomes part of your project history and is essential for team collaboration.

git config --global user.name "Your Name"
git config --global user.email "your@email.com"

# Verify configuration
git config --list
$ git config --global user.name "Kalyan"
$ git config --global user.email "kalyan@example.com"
$ git config --list
user.name=Kalyan
user.email=kalyan@example.com

✅ Every commit you make will be attributed to this identity.
4️⃣ Create First Git Repository (git init)

📌 What is git init?

git init creates a hidden .git folder in your project, turning it into a Git repository. This folder stores all version history, branches, and configuration.

mkdir my-first-project
cd my-first-project
git init
# Output: Initialized empty Git repository in /my-first-project/.git/
$ mkdir my-first-project
$ cd my-first-project
$ git init
Initialized empty Git repository in /my-first-project/.git/

✅ A hidden .git folder is created — this is your repository!
🔍 To see it: ls -la (Mac/Linux) or dir /a (Windows)
5️⃣ Check Status (git status)

📌 What is git status?

git status shows the current state of your working directory and staging area — which files are modified, staged, or untracked. Use it frequently!

git status
# Shows: untracked files, modified files, staged files
$ git status
On branch main

No commits yet

Untracked files:
  index.html
  style.css

nothing added to commit but untracked files present

✅ Use 'git add' to start tracking files.
6️⃣ Staging Area (git add) — Prepare Changes

📌 What is Staging?

The staging area is like a "draft" — you can review changes before they become permanent commits. git add moves changes to staging.

# Add specific file
git add index.html

# Add all files in current directory
git add .
Working Directory (modified files)
↓ git add
Staging Area (files ready to commit)
↓ git commit
Repository (saved permanently)
✅ git add . stages all changes in current directory
✅ You can unstage with 'git reset'
💡 Always check 'git status' after adding!
7️⃣ Commit — Save Snapshot (git commit)

📌 What is a Commit?

A commit is a snapshot of your project at a specific point in time. Each commit has a unique ID and a message describing what changed.

git commit -m "Initial commit: added homepage structure"

# View commit history
git log
git log --oneline
$ git commit -m "Initial commit: added homepage structure"
[main (root-commit) a1b2c3d] Initial commit: added homepage structure
2 files changed, 45 insertions(+)
create mode 100644 index.html
create mode 100644 style.css

$ git log --oneline
a1b2c3d Initial commit: added homepage structure

✅ Good commit messages describe WHAT and WHY, not HOW.
8️⃣ Complete Git Workflow (Real Developer Steps)
# 1. Create/Modify files
echo "Hello" > index.html

# 2. Check what changed
git status

# 3. Stage changes
git add .

# 4. Commit with meaningful message
git commit -m "Added homepage content"

# 5. View history
git log --oneline
⚡ Developer Workflow Simulation:

1. Edit files in your code editor
2. $ git status → shows what changed
3. $ git add . → stage all changes
4. $ git commit -m "Describe your changes" → save snapshot
5. $ git log → see history

🔁 Repeat: edit → add → commit → (then push to GitHub later)

📝 Good Commit Messages vs Bad

✅ Good examples

"Added login form validation"
"Fixed navbar responsiveness on mobile"
"Refactored API service into separate module"

❌ Bad examples

"update"
"fix stuff"
"changes"
"asdf"

⚠️ Common Beginner Mistakes & Solutions
  • ❌ Forgetting git add . before commit → nothing saved. Fix: Always run git status before committing.
  • ❌ Vague commit messages → "stuff" is not helpful. Fix: Write descriptive messages that explain WHY.
  • ❌ Not checking git status before committing → missing changes. Fix: Make git status a habit.
  • ❌ Committing unnecessary files (node_modules, .env) → repo becomes huge. Fix: Use .gitignore (learn tomorrow).
  • ❌ Confusing staging area vs commit. Fix: Staging = "I want to save these changes", Commit = "I'm saving them now".

💼 Common Git Interview Questions

❓ What is Git?

Distributed version control system that tracks changes in source code during development.

❓ What is a repository?

A folder managed by Git containing project files and complete version history.

❓ Difference between git add and git commit?

git add stages changes (prepares them), git commit permanently saves them to history.

❓ What is a commit hash?

A unique identifier (like a1b2c3d) for each commit, used to reference specific snapshots.
