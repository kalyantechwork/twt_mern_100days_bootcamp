1️⃣ What are Branches? (Parallel Development)

📌 Purpose & Usage

Branches are separate versions of your code. Think of them as creating a copy where you can experiment without affecting the main project. This allows multiple developers to work on different features simultaneously without interfering with each other.

Where it's used: Every professional software team — feature development, bug fixes, hotfixes, experimentation, release preparation.

🌿 Branch Diagram:
main ─────────────────────────────────▶
│
└── feature-navbar ──▶ (changes) ──▶ merge back to main
└── bugfix-header ───▶ (changes) ──▶ merge back to main
2️⃣ Why Developers Use Branches

✅ Isolate Features

Work on new features without breaking main code.

🐛 Bug Fixes

Fix bugs separately and merge when ready.

👥 Team Collaboration

Multiple developers work simultaneously on different branches.

🔀 Experiment Safely

Try new ideas — if it fails, delete the branch.

3️⃣ Essential Branch Commands
# List all branches
git branch

# Create a new branch
git branch feature-login

# Switch to a branch
git checkout feature-login

# Create & switch in one command
git checkout -b feature-navbar

# Delete a branch (after merging)
git branch -d feature-login
$ git branch
* main
$ git checkout -b feature-login
Switched to a new branch 'feature-login'
$ git branch
* feature-login
main
$ git checkout main
Switched to branch 'main'
$ git branch -d feature-login
Deleted branch feature-login (was a1b2c3d)

✅ 'git branch -a' shows all local and remote branches
4️⃣ Merging Branches (Combine Code)

📌 What is Merging?

Merging combines changes from one branch into another. Typically, you merge feature branches into main when the feature is complete and tested.

# First, switch to the branch you want to merge INTO (e.g., main)
git checkout main

# Merge the feature branch into main
git merge feature-login

# Delete the feature branch after merge (optional)
git branch -d feature-login
main: A---B---C---F---G
\
feature: D---E (merged into main at F)
$ git checkout main
$ git merge feature-login
Updating a1b2c3d..e4f5g6h
Fast-forward
index.html | 2 +-
1 file changed, 1 insertion(+), 1 deletion(-)

✅ Fast-forward merge (no merge commit) — happens when branch is ahead of main.
5️⃣ Pull Requests (PR) — GitHub Collaboration

📌 What is a Pull Request?

A Pull Request is a request to merge your branch into another branch (usually main). It allows code review, discussion, and automated testing before merging — essential for team workflows.

# Step 1: Push your branch to GitHub
git push origin feature-contact-page

# Step 2: Go to GitHub.com → Your repository
# Step 3: Click "Pull Requests" → "New Pull Request"
# Step 4: Select base: main, compare: feature-contact-page
# Step 5: Click "Create Pull Request"
# Step 6: Team reviews → Click "Merge Pull Request"
Local → git push → GitHub → Open PR → Review → Merge → main updated ✅
🔄 Pull Request Workflow Summary:
1. git push origin feature-branch
2. Open GitHub → "Compare & Pull Request"
3. Write description, add reviewers
4. Team reviews code, leaves comments
5. After approval, click "Merge Pull Request"
6. Delete the branch (optional)

💡 PRs are the standard for code review in every tech company!
6️⃣ Merge Conflicts (When Git Can't Auto-Merge)

📌 What is a Merge Conflict?

A merge conflict occurs when two branches modify the same line(s) of code, or one branch deletes a file while another modifies it. Git cannot automatically decide which change to keep — you must resolve it manually.

# When merging, Git shows:
Auto-merging index.html
CONFLICT in index.html
Fix conflicts and then commit.

# Inside conflicted file:
<<<<<<< HEAD
Hello from main branch
=======
Hello from feature branch
>>>>>>> feature-branch
📝 How to Resolve Merge Conflicts:
1. Open the conflicted file
2. Look for conflict markers: <<<<<<< HEAD, =======, >>>>>>>
3. Edit the file to keep the correct code
4. Remove all conflict markers (<<<<<<, =======, >>>>>>)
5. Save the file
6. git add .
7. git commit -m "Resolved merge conflict"
8. git push

💡 Merge conflicts are normal — don't panic! Practice resolving them.
7️⃣ Branching Best Practices

📌 Name branches clearly

feature/login-page, bugfix/navbar-crash, hotfix/security-patch

🔀 Keep main branch stable

Never commit directly to main — always use PRs

🗑️ Delete merged branches

Keeps repository clean and organized

🔄 Pull before pushing

git pull origin main to avoid conflicts

8️⃣ Real Developer Team Workflow (GitHub Flow)
main branch (production-ready)
│
├── feature/auth (Developer A)
├── feature/payment (Developer B)
└── bugfix/ui (Developer C)
Each developer → creates PR → code review → merge → deploy
# Developer A's day:
git checkout -b feature/api-integration
# ... write code ...
git add .
git commit -m "Integrated payment API"
git push origin feature/api-integration
# Then open Pull Request on GitHub

# Developer B reviews PR, approves, merges
# All developers: git pull origin main
✅ This is the exact workflow used at Google, Microsoft, Amazon, and every tech company.
🔁 Daily cycle: pull main → create branch → work → push → PR → merge → repeat
⚠️ Common Beginner Mistakes
  • ❌ Forgetting to switch branch before making changes → commits on wrong branch. Fix: git checkout to correct branch.
  • ❌ Merging without pulling latest main → creates unnecessary conflicts. Fix: git pull origin main before merging.
  • ❌ Deleting branch before merging → losing work. Fix: Only delete after confirming merge.
  • ❌ Not pushing feature branch to GitHub → no PR can be created. Fix: git push origin branch-name.
  • ❌ Fear of merge conflicts → conflicts are normal! Practice resolving them.

💼 Common Interview Questions

❓ What is a Git branch?

Separate line of development — allows parallel work without affecting main code.

❓ What is a Pull Request?

Request to merge one branch into another — enables code review and collaboration.

❓ How to resolve merge conflicts?

Manually edit conflicted files, remove conflict markers, then git add & commit.

❓ How to see all branches?

git branch (local) or git branch -a (including remote)
